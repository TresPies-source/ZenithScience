**Date:** 2026-02-20

## Grounding in the Actual Codebase

Before reasoning, the following were confirmed by reading source:

- `AgentChat.svelte` calls `createAgent(projectId, 'Research Agent', activeProvider)` on every
  `onMount`. If the gateway returns 404 on a stale `agent_id` it re-creates again. Agent IDs are
  ephemeral; none are stored anywhere in SQLite.
- `GatewayClient.create_agent()` sends `{ workspace_root, active_mode }` to
  `POST /v1/gateway/agents`. No `document_id` or `session_id` is included in the payload.
- `AgentResponse` has `tool_calls: Vec<ToolCall>` and `finish_reason: Option<String>` but both are
  commented as "NOT sent by the gateway." They are dead fields today.
- `CompletionRequest.Tools` and `CompletionResponse.ToolCalls` exist in Go types but are
  described as "already exists, ignored by Ollama provider currently."
- `update_section_content` and `update_document_content` are both live Tauri commands. They write
  directly to SQLite — no gateway involvement. This is the write-back entry point.
- `Document` model has: `id`, `owner_id`, `mode` (Draft/Structured), `sections: Vec<Section>`.
  Each `Section` has `id`, `document_id`, `title`, `content`, `owner_id`, `order`, `locked`.
- Memory system is live: `POST /v1/memory` and `POST /v1/memory/search` both work, confirmed by
  integration tests in `gateway/client.rs`.
- Gateway agent map is confirmed in-memory (no SQLite persistence on gateway side).

## QUESTION 1: Agent Persistence & Document Readability

### Tension Statement

The gateway's agent store is in-memory and volatile. Every app launch destroys session continuity.
The real question is whether "continuity" must live in the agent identity itself, or whether it can
be reconstructed from other durable sources that already exist (SQLite document store + gateway
memory system).

---

### Route A — Document-Anchored Agent Identity

Store a `agent_id` field on the SQLite `Document` row. At session start, if the document already
has an `agent_id`, attempt to resume it; if the gateway returns 404, recreate and update the
stored ID. Chat history is stored in SQLite keyed by `document_id`.

**Upside:**
- True session continuity. One agent per document aligns semantically with the academic writing
  model (each document has its own persistent collaborator).
- Chat history in SQLite is durable, searchable, and survives gateway restarts.
- Document context injection at session start is straightforward: query SQLite for full Document +
  sections, inject as system prompt.

**Downside:**
- Gateway agent configs are in-memory; the stored `agent_id` becomes stale on every gateway
  restart (which happens on every app launch since it's a sidecar). This means the "resume"
  attempt always fails and we always re-create — the stored ID is worthless in practice for a
  sidecar architecture.
- Requires a schema migration: add `agent_id` column to `documents` table in SQLite.
- Adds a new SQLite table for chat history. Not trivial — requires Tauri command surface changes.
- The constraint "no gateway Go source changes" means we cannot add persistence to the gateway's
  agent map. The agent map will always reset on restart.

**Verdict:** This route is architecturally correct but is undermined by the sidecar restart cycle.
The agent_id stored in SQLite will be stale on every launch. The benefit of "same agent" is
illusory — you'd always be creating a new agent with the old ID discarded.

---

### Route B — Memory-as-History (Gateway Memory as Continuity Layer)

No agent persistence. Agents stay ephemeral and are created fresh per session. The continuity
layer is the gateway memory system (`POST /v1/memory`), which is already built, tested, and
durable (backed by the gateway's own persistence, not session memory).

On new session:
1. Create fresh agent.
2. Search memory for entries tagged with this `document_id`: `POST /v1/memory/search` with
   query `"document:{document_id}"`.
3. Inject top-N relevant memory items into the system prompt alongside the full document structure.
4. As the session progresses, important exchanges are saved to memory (the "Save to memory" button
   already exists in `AgentChat.svelte` — just needs to tag entries with `document_id`).

Document context = structured SQLite query (title, mode, sections[{title, content, owner_id}])
formatted as a system prompt preamble.

**Upside:**
- Works perfectly with the sidecar restart reality. Memory survives gateway restarts because it
  is stored durably in the gateway's own backing store.
- Zero schema migration. Zero new Tauri commands needed.
- The "Save to memory" button already exists in `AgentChat.svelte` — the user interaction model
  is already there, just needs `document_id` tagging.
- Memory is semantic and searchable — future sessions can retrieve the most relevant prior
  context, not just the last N messages.
- Leverages already-built infrastructure (`search_memory`, `store_memory` in `GatewayClient`).

**Downside:**
- Memory retrieval is semantic search, not ordered chat replay. The agent does not see the exact
  prior conversation verbatim, only distilled memories. This is a different UX than "resume
  where you left off."
- If the gateway memory system's backing store is itself in-memory (this needs verification), this
  route also loses data on restart.
- Requires discipline from users to save important exchanges — automatic saving risks memory bloat.
- Memory items are unstructured text; structured document context (sections, ownership, mode) must
  be injected separately each session anyway.

**Verdict:** Strong route for semantic continuity. The "Save to memory" affordance already exists.
The critical unknown is whether the gateway's memory store is durable across restarts.

---

### Route C — Project Agent + Document Resource Model

One persistent agent per project (not per document). The Svelte agent store lifts from
`document_id` scope to `project_id` scope. Documents become resources the agent can read on
demand.

This requires a new gateway endpoint: `GET /v1/gateway/documents/:id` that the agent can call as
a tool to fetch a document's structured content. The agent is project-wide and documents are
just inputs to it.

**Upside:**
- An agent that knows the whole project (multiple documents, bibliography, references) is more
  powerful than one that knows only one document.
- Agent creation is rarer (once per project session) — less overhead.

**Downside:**
- Gateway Go source changes required (new endpoint). This violates the stated constraint.
- The academic writing model is document-centric, not project-centric. Cruz's use case is "help
  me write this paper" not "help me manage my research portfolio."
- Context window pressure: a project-scoped agent's system prompt grows with every document.
  Local Ollama models with limited context windows will degrade significantly.
- Significantly more complex Svelte state management.

**Verdict:** Wrong scope for academic writing. Project scope is a v2 idea. Rejected for v1.1.

|:---------------------------------|:----------------------:|:---------------------------:|:-----------------------:|
| Survives gateway sidecar restart | No (agent ID stale)    | Yes (if memory is durable)  | No                      |
| Schema migration needed          | Yes                    | No                          | Yes                     |
| New Tauri commands needed        | Yes (chat history)     | Minimal (tag existing)      | Yes                     |
| Gateway Go changes needed        | No                     | No                          | Yes (violates constraint)|
| Document context richness        | High                   | High (injected each session)| High                    |
| Semantic continuity              | Low (ephemeral agent)  | High (semantic memory)      | Medium                  |

| Fits local-first sidecar model   | No                     | Yes                         | No                      |

---

### Recommended Direction — Question 1

**Route B (Memory-as-History) with structured document injection on every session.**

Rationale: The gateway sidecar restarts on every app launch. Any approach that tries to make the
agent identity persistent across restarts is fighting the architecture. Route B accepts this
reality and uses the memory system — which already exists and is tested — as the continuity layer.

The recommended implementation shape:

1. On session start, call `create_agent` fresh (as today). No change to gateway.
2. Before the first user message, perform a `search_memory` call with query constructed from
   document metadata (e.g., document title + "document:" + document_id as a tag), retrieve
   top-5 relevant memory items.
3. Inject a structured system prompt: document title, mode, sections array (title + content +
   owner), followed by the retrieved memory items as "prior context."
4. Tag all `store_memory` calls with `metadata: { document_id: "..." }` so future sessions can
   filter them.
5. Auto-save the final assistant message in each chat turn to memory (the gateway already supports
   this; the button already exists — just make it automatic for turns where the agent produced
   substantive content, and manual for exploratory turns).

The one risk to verify: confirm that the gateway's memory backing store persists across
gateway process restarts. If it does not, Route B degrades to ephemeral context too, and the
correct fallback is to store memory items in a SQLite table in the Tauri layer and inject them as
a constructed context string — bypassing the gateway memory search entirely.

## QUESTION 2: Provider-Aware Tool Calling

### Tension Statement

The gateway's `CompletionRequest.Tools` and `CompletionResponse.ToolCalls` fields already exist
but are never populated. The `TOOL_CALL:` text format works universally but is fragile. The
question is where normalization lives: at the provider boundary, at the gateway loop level, or
separated into planning vs. execution phases.

---

### Route A — Provider Capability Declaration

Each `ModelProvider` implements `SupportedCapabilities() []string`. The gateway loop checks
capabilities before choosing tool call strategy. Providers that declare `"native_tools"` get
`Tools[]` in `CompletionRequest` and return `ToolCalls[]` in `CompletionResponse`. Others get
the `TOOL_CALL:` text format injected into the system prompt.

**Upside:**
- Gateway loop is unified — same code path regardless of provider capability.
- Graceful degradation is explicit and declared.
- Adding a new provider is straightforward: implement capability list.

**Downside:**
- Requires gateway Go source changes (add method to `ModelProvider` interface). This violates
  the constraint.
- The gateway's tool loop must branch on capability at runtime. Two code paths for tool
  parsing must coexist in the gateway.
- Model-level capability (not just provider-level) matters: Ollama supports tools on
  qwen3/llama3.1 but not on all models. Provider-level declaration is too coarse.

**Verdict:** Conceptually clean but requires Go changes and model-level capability is still not
addressed.

---

### Route B — Two-Phase Tool Architecture (Planning + Execution)

Phase 1: Use the best available model to produce a structured JSON tool plan. The planning prompt
asks for a JSON array of `{tool, args}` pairs, not free-text `TOOL_CALL:` markers.

Phase 2: Gateway executes the plan directly without another LLM call. Tool results are collected
and either returned to the user or fed back for synthesis.

**Upside:**
- Provider-agnostic: any model that can produce JSON output can participate in planning.
- Execution is deterministic — no parsing of free-text LLM output for tool invocations.
- Separation of concerns is clean.

**Downside:**
- Requires gateway Go changes to implement the two-phase loop. Violates constraint.
- JSON output mode is not universal — small local Ollama models often fail structured JSON.
- Loses the "reasoning about tool results" capability: the agent cannot see tool output and
  decide what to do next. The plan is fixed at Phase 1.
- Two-pass architecture increases latency by at least one full LLM call per tool-using turn.

**Verdict:** Interesting architecture for batch workflows (like the existing DAG pipeline), but
wrong for interactive document chat where the agent must react to tool results.

---

### Route C — Normalize at the Provider Layer

Each provider's `GenerateCompletion` implementation handles tool calling internally and returns a
normalized `ToolCalls[]` in `CompletionResponse`. The gateway tool loop is purely normalized:
send request with `Tools[]`, receive `ToolCalls[]`, execute, repeat.

- Anthropic provider: sends `tools` array in API request, parses `tool_use` content blocks,
  returns normalized `ToolCalls[]`.
- OpenAI provider: sends `functions`/`tools` array, parses `function_call`/`tool_calls`, returns
  normalized `ToolCalls[]`.
- Ollama provider: probes model capability (by model name prefix or a capability check call),
  uses `tools` field in `/api/chat` if supported; if not, falls back to `TOOL_CALL:` text parsing
  but still returns `ToolCalls[]` — just extracted differently.
- `TOOL_CALL:` text format is kept only as the Ollama fallback; it never leaks into the gateway
  loop.

**Upside:**
- Gateway tool loop becomes clean and normalized. No branching in the loop itself.
- The existing `Tools[]` and `ToolCalls[]` fields in `CompletionRequest`/`CompletionResponse`
  are exactly the right abstraction — they just need to be wired up.
- Ollama model capability can be detected by model name pattern (`qwen`, `llama3.1`, etc.) or by
  attempting a tool call and falling back on error.
- No new interface methods required — `GenerateCompletion` already accepts `Tools[]` and is
  expected to return `ToolCalls[]`.
- Constraint-compatible: provider implementations are Node.js MCP server level changes (acceptable
  per Cruz's preferences) for the Ollama fallback; Anthropic/OpenAI providers are pure Go
  inside the existing interface.

**Downside:**
- Requires gateway Go source changes to wire up `Tools[]` in the Anthropic and OpenAI providers,
  and to add model-name-based capability detection in the Ollama provider. Violates the "no
  gateway Go changes" constraint as stated — but this constraint may need reexamination for Q2.
- Ollama model name-based detection is brittle. A model named `custom-qwen-finetuned` might
  support tools but not be detected.
- The text-format fallback in Ollama still requires a parser somewhere.

**Verdict:** The architecturally strongest route. The abstraction already exists in the Go types —
it just needs to be wired. The gateway tool loop becomes a clean normalize-execute-normalize cycle.

---

### Tradeoff Comparison — Question 2

| Criterion                        | Route A (Capability Declaration) | Route B (Two-Phase) | Route C (Normalize at Provider) |
|:---------------------------------|:--------------------------------:|:--------------------:|:-------------------------------:|
| Requires gateway Go changes      | Yes                              | Yes                  | Yes (but minimal, in provider impl) |
| Gateway loop complexity          | Medium (two strategies)          | Low (no LLM in exec) | Low (normalized everywhere)     |
| Provider-agnostic execution      | Partial                          | Yes                  | Yes                             |
| Interactive tool-use support     | Yes                              | No (plan is fixed)   | Yes                             |
| Model-level capability handling  | No (provider-level only)         | N/A                  | Yes (Ollama name probe)         |
| Uses existing Go types           | Partial                          | No                   | Yes (CompletionRequest.Tools)   |
| Graceful degradation             | Yes                              | Yes (JSON fallback)  | Yes (Ollama text fallback)      |

---

### Recommended Direction — Question 2

**Route C (Normalize at the Provider Layer).**

Rationale: The Go interface already has the correct shape. `CompletionRequest.Tools` and
`CompletionResponse.ToolCalls` were designed for this. Wiring them up in each provider
implementation is the minimum-delta path. The gateway tool loop should not need to know which
provider it is talking to — that is the entire point of the `ModelProvider` interface.

The "no gateway Go changes" constraint should be relaxed for Q2. The changes needed are not
structural — they are filling in already-stubbed behavior in existing provider implementations.
This is categorically different from adding new gateway endpoints or changing the agent loop's
architecture.

Recommended implementation shape:

1. Anthropic provider: populate `request.tools` from `CompletionRequest.Tools`, parse
   `tool_use` blocks in the response, return `CompletionResponse.ToolCalls`.
2. OpenAI provider: same pattern for `tool_calls` in the response.
3. Ollama provider: check model name for known tool-capable patterns; if match, use `tools` field
   in the chat request; if response contains `tool_calls`, parse them; otherwise fall back to
   scanning the `message.content` for `TOOL_CALL:` markers and return as normalized `ToolCalls[]`.
4. The gateway tool loop: send `Tools[]` with every request that has registered tools; if
   response contains `ToolCalls[]`, execute them; if not, treat the response as a final reply.

This removes `TOOL_CALL:` from the system prompt for capable providers while keeping it as an
Ollama-only text-mode fallback that is invisible to the gateway loop.

## QUESTION 3: Provider-Aware Document Write-Back

### Tension Statement

The academic writer needs precision: wrong content written to the wrong section corrupts their
work. The gateway needs to produce a usable patch from LLM output that varies in structure from
perfectly compliant JSON (Claude) to undifferentiated prose (small local models). The question is
whether enforcement, extraction, or human confirmation is the right control point.

---

### Route A — Structured Output Enforcement

For capable providers: require `DOCUMENT_PATCH: {json}` in the response. For incapable ones:
treat the entire response as content for a catch-all "Agent Draft" section. User applies it
manually.

**Upside:**
- Low latency — no second LLM call.
- Capable providers (Anthropic, GPT-4) comply reliably with structured output instructions.
- Clear human escape hatch for weak models.

**Downside:**
- For Ollama, especially small models, compliance is unreliable even with strong prompting. A
  small model asked to write an introduction will often just write it, not format it as JSON.
- The "Agent Draft" fallback for weak models means write-back is effectively non-functional for
  the primary local-first use case (Ollama). This route implicitly punishes local model users.
- "Agent Draft" section accumulates unstructured content; the user must manually copy-paste to
  the right section — this defeats the purpose of agent-assisted writing.

**Verdict:** Works well for API providers; fails for the local-first Ollama default. Inadequate
for the stated philosophy.

---

### Route B — Post-Processing Extraction (Two-Pass)

After the primary response, run a small fast extraction call: "Given this agent response, produce
JSON: `{section_title, action, content}`". The extraction model is fast (can be a small Ollama
model like llama3.2:1b) and does a narrow structured task.

**Upside:**
- Works regardless of primary model capability — the extraction call is separate and narrow.
- The primary response remains natural and readable to the user.
- Extraction models are small and fast; latency impact is acceptable for local Ollama.

**Downside:**
- Two LLM calls per write-back turn. In a streaming UX, this means the user waits for a second
  call after the response is complete.
- Extraction can fail for ambiguous responses: "Here's an introduction" — which section?
- Requires a second LLM API call, which means the gateway needs a secondary model configured for
  extraction. If only one Ollama model is running, this is the same model doing two calls.
- Extraction adds complexity to the gateway's agent loop — requires gateway Go changes.

**Verdict:** Technically sound for the write-back problem but adds latency and complexity. Better
suited for batch document generation than interactive chat.

---

### Route C — Patch-by-Intent with Human Confirmation

The agent never writes directly to the document. Instead, the agent response includes a
`patch_intent` signal:

```
I would like to draft the Introduction section. Here is my proposal:

[prose content]

---
PATCH_INTENT: {"section_title": "Introduction", "action": "create", "preview": true}
```

The gateway parses the `PATCH_INTENT:` marker (simpler than `TOOL_CALL:` — just a structured
footer) and returns it in the `AgentResponse`. The Svelte frontend detects the intent, renders a
preview diff showing what would change, and presents "Apply" / "Discard" buttons. If the user
clicks Apply, the frontend calls `update_section_content` directly — the Tauri command that
already exists and is tested.

**Upside:**
- Human always confirms. For academic writing where precision matters, this is not a limitation —
  it is the correct design. The writer owns their document; the agent proposes.
- `update_section_content` is already a live Tauri command. No new backend code needed.
- `PATCH_INTENT:` parsing is much simpler than `TOOL_CALL:` — it is a structured footer on the
  response, not embedded in the text.
- Works with any model capability. Even a weak Ollama model can produce prose + a footer marker.
  The agent is not being asked to produce JSON structure — just to end its response with a simple
  declaration of intent.
- The user sees the prose reasoning and the proposed change together. This supports the "AI as
  thinking partner" philosophy stated in the architecture overview.
- No second LLM call. No gateway changes needed beyond returning `patch_intent` in the response.

**Downside:**
- An extra confirm step. If the agent is confident and correct, this adds friction.
- `PATCH_INTENT:` marker still requires the model to produce it. Weak models may omit it entirely,
  degrading to no write-back.
- Section targeting ("Introduction") is still approximate — model must name the section correctly.
  If the document has sections with similar titles, ambiguity is possible.
- This is not automation — it is augmentation. If the use case evolves toward "auto-fill my
  methodology section," Route C will not satisfy it.

**Verdict:** Best fit for academic writing in v1.1. The confirm step is a feature, not a bug.

### Recommended Direction — Question 3

**Route C (Patch-by-Intent with Human Confirmation).**

Rationale: The document write-back problem is not primarily a technical challenge — it is a trust
challenge. An academic researcher must trust that nothing is written to their document without
their knowledge. Route C makes trust explicit by requiring human confirmation. The confirm step
also surfaces the agent's reasoning (the prose) alongside the intent, letting the writer evaluate
the proposal in context.

The existing `update_section_content` Tauri command is the right write-back path. It already
handles Draft and Structured mode. The Svelte layer needs to detect `patch_intent` in the
`AgentResponse` and render a diff preview component — this is a frontend-only change.

The `PATCH_INTENT:` footer is intentionally simpler than `TOOL_CALL:`. It does not need to carry
content — only the intent signal. The content is already in the response prose visible to the user.
When the user clicks Apply, the frontend uses the prose content as the section payload.

## How the Three Recommendations Interact

The three questions are not independent. Here is how the recommended directions compound:

**Q2 (Route C — Normalize at Provider) enables Q3 (Route C — Patch-by-Intent):**
When native tool calling is available (Anthropic, capable Ollama), the agent can call a
`propose_section_write` tool that returns a structured `patch_intent`. This is more reliable than
a text footer because the tool call is structured. For providers without native tools, the text
footer fallback applies. Route C on Q2 gives Route C on Q3 a reliable high path (tool-based intent)
and a graceful low path (text footer).

**Q1 (Route B — Memory-as-History) and Q3 (Patch-by-Intent) reinforce each other:**
When the user clicks Apply to write content to a section, that write-back event itself can be
saved to memory: "On 2026-02-20, agent proposed and user accepted an Introduction section of 400
words about X." This becomes the richest possible memory item — not just a conversation excerpt
but a confirmed authorship event. Future sessions recover not just what was discussed, but what
was actually written and accepted.

**Q1 (Memory) and Q2 (Tool normalization) create a future capability:**
Once tool calling is normalized (Q2), a `search_memory` tool can be registered and called by the
agent during its reasoning loop. The agent can proactively retrieve its own prior work on a
document mid-conversation without a system prompt injection. This is the long-term payoff of
doing Q2 correctly.

**The dependency order for implementation is therefore:**
1. Q3 first (Patch-by-Intent) — frontend only, no gateway changes, highest user value.
2. Q1 second (Memory tagging) — minimal changes, leverages existing `store_memory`.
3. Q2 last (Tool normalization) — requires gateway Go changes, but unlocks the full capability
   stack including tool-based intent signals and memory search as a tool.

## Open Items to Verify

1. **Gateway memory durability:** Is the gateway's memory backing store durable across process
   restarts? If not, Route B on Q1 requires a SQLite fallback in the Tauri layer.
2. **Ollama tool-capable model names:** What model name patterns should the Ollama provider use
   to probe tool support? Confirm `qwen`, `llama3.1`, `llama3.2` as starting patterns.
3. **Gateway tool loop current state:** Does the gateway currently execute tool calls from
   `CompletionResponse.ToolCalls`, or does it only parse `TOOL_CALL:` text? This determines
   how much of the Q2 gateway-side work is additive vs. replacement.
